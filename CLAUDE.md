# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

BuckOS is a Buck2-based Linux distribution build system inspired by Gentoo's ebuild system. Each Linux package is a reproducible Buck2 target with declared source, build configuration, USE flags, and dependencies. The build pipeline produces individual packages, root filesystems, kernels, and bootable images.

## Build Commands

```bash
# Build an individual package
buck2 build //packages/linux/core/bash:bash
buck2 build //packages/linux/system/libs/crypto/openssl:openssl

# Build for the linux target platform (default is //platforms:linux-x86_64;
# the hermetic target platform is //platforms:linux-target, and a faster
# host-toolchain variant is //platforms:linux-target-host)
buck2 build //packages/linux/editors/entr:entr \
    --target-platforms //platforms:linux-target-host

# List targets
buck2 targets //packages/linux/system/...

# Build the full rootfs
buck2 build //packages/linux/system:buckos-rootfs

# Build the everything-meta target
buck2 build //:complete

# Build and boot a rootfs in QEMU
buck2 build //packages/linux/system:qemu-boot --show-output
```

If the Buck2 daemon is slow to start after a `.bzl` edit (it can take 90s+),
extend the startup window:

```bash
BUCKD_STARTUP_TIMEOUT=600 buck2 build //packages/linux/system:buckos-rootfs
```

Do not `buck2 killall` proactively — restarting the daemon costs ~90s and
invalidates in-memory caches.

### Invalidating Stale Materializer Cache

If files were deleted from `buck-out/` out-of-band and the materializer is stale (builds fail with "No such file or directory" for generated scripts), fix individual targets without `buck2 killall`:

1. **For `export_file` targets** (e.g. helper scripts checked into `defs/scripts/`):
   Manually copy the source into the expected buck-out path:
   ```bash
   mkdir -p buck-out/v2/gen/root/<HASH>/defs/scripts/__<target-name>__/
   cp defs/scripts/<source-file>.sh buck-out/v2/gen/root/<HASH>/defs/scripts/__<target-name>__/<target-name>
   chmod +x buck-out/v2/gen/root/<HASH>/defs/scripts/__<target-name>__/<target-name>
   ```

2. **For `ctx.actions.write()` outputs** (e.g. scripts generated by the rules in
   `defs/rules/`): add or change a comment in the script content so the action
   hash changes and Buck2 regenerates the file.

3. **General rule**: never delete directories under `buck-out/v2/gen/root/` — they
   contain shared build scripts used by all packages.

## Architecture

### Core build definitions (`defs/`)

The build is layered as:

```
packages/linux/.../BUCK   (user surface)
   |
   v
defs/packages/<lang>.bzl  (thin wrappers, choose build_rule)
   |
   v
defs/package.bzl          (package() macro: source download, patch merge,
                           USE flag expansion, host-tool injection,
                           transform chain, alias)
   |
   v
defs/rules/<lang>.bzl     (per-build-system rules: <lang>_build)
defs/rules/_common.bzl    (shared attrs, phase helpers)
```

Files of interest under `defs/`:

- `package.bzl` — the `package()` macro every wrapper calls into.
- `packages/*.bzl` — one-line wrappers per build system (see below).
- `rules/*.bzl` — Buck2 rules per build system; `_common.bzl` defines shared
  attrs and phase functions.
- `use_helpers.bzl` — USE-flag → `select()` helpers (`use_bool`, `use_dep`,
  `use_configure_arg`, `use_feature`, `use_expand_*`, `use_versioned_dep`).
- `empty_registry.bzl` — `PATCH_REGISTRY = {}` fallback consumed by
  `package()`; replace the load line in `defs/package.bzl` with
  `//patches:registry.bzl` to use a private registry.
- `host_tools.bzl`, `toolchain_helpers.bzl` — toolchain plumbing.
- `package_sets.bzl` — system profile aggregations (minimal, server, desktop,
  developer, …).
- `download.bzl`, `keys/`, `vendor_sources.bxl` — source download + mirror
  glue.

### Package layout

Packages live in `packages/linux/<category>/<name>/BUCK`. A typical BUCK file:

1. Loads one or more wrappers from `//defs/packages:<lang>.bzl`.
2. Calls the wrapper(s) with `name`, `version`, `url`, `sha256`, and any
   build-system-specific kwargs.

A real example — `packages/linux/core/bash/BUCK`:

```python
load("//defs/packages:autotools.bzl", "autotools_package")

autotools_package(
    name = "bash",
    version = "5.3",
    url = "https://mirrors.kernel.org/gnu/bash/bash-5.3.tar.gz",
    sha256 = "0d5cd86965f869a26cf64f4b71be7b96f90a3ba8b3d74e27e8e9d9d5550f31ba",
    description = "The GNU Bourne Again SHell",
    homepage = "https://www.gnu.org/software/bash/",
    license = "GPL-3.0",
    use_deps = {
        "readline": [
            "//packages/linux/system/terminal/readline:readline",
            "//packages/linux/system/terminal/ncurses:ncurses",
        ],
    },
    use_configure = {
        "nls":      ("--enable-nls",      "--disable-nls"),
        "readline": ("--with-installed-readline", "--without-installed-readline"),
    },
    configure_args = ["--without-bash-malloc"],
    post_install_cmds = ["""ln -sf bash "$DESTDIR/usr/bin/sh" """],
)
```

A single BUCK file may load multiple wrappers when it defines a mix of
packages with different build systems — see
`packages/linux/system/security/encryption/BUCK` for a file that uses
`autotools_package`, `binary_package`, `cargo_package`, `cmake_package`,
`go_package`, and `meson_package` side by side.

### Package wrappers (`defs/packages/`)

Each wrapper is a three-line shim that delegates to `package()` with a fixed
`build_rule`. Use the wrappers in BUCK files; `package()` itself is the
escape hatch for cases that need to choose `build_rule` dynamically.

| Load line | Macro(s) | Use case |
|---|---|---|
| `//defs/packages:autotools.bzl` | `autotools_package`, `make_package` | `./configure && make && make install`; `make_package` is the same but defaults `skip_configure = True` |
| `//defs/packages:binary.bzl` | `binary_package` | Custom `install_script` for prebuilt or off-script packages |
| `//defs/packages:cmake.bzl` | `cmake_package` | CMake projects (built with Ninja) |
| `//defs/packages:meson.bzl` | `meson_package` | Meson projects (built with Ninja) |
| `//defs/packages:cargo.bzl` | `cargo_package` | Rust / Cargo |
| `//defs/packages:go.bzl` | `go_package` | Go |
| `//defs/packages:python.bzl` | `python_package` | Python (pip or setup.py) |
| `//defs/packages:perl.bzl` | `perl_package` | `perl Makefile.PL && make && make install` |
| `//defs/packages:mozbuild.bzl` | `mozbuild_package` | Firefox / mach-based builds |

### Rule-specific kwargs

In addition to the common attrs (`name`, `version`, `url`, `sha256`,
`description`, `homepage`, `license`, `cpe`, `libraries`, `deps`,
`host_deps`, `patches`, `configure_args`, `extra_cflags`, `extra_ldflags`,
`pre_configure_cmds`, `post_install_cmds`, `env`, `use_deps`,
`use_configure`, `use_features`, `use_transforms`, `transforms`,
`exclude_patterns`, `strip_components`, `format`, `local_only`,
`filename`), each wrapper forwards the following to its underlying
`<lang>_build` rule:

| Wrapper | Rule-specific kwargs |
|---|---|
| `autotools_package` | `configure_prefix_deps`, `configure_script`, `skip_configure`, `cc_as_configure_arg`, `skip_cc_auto_arg`, `skip_host_arg`, `build_subdir`, `pre_build_cmds`, `make_args`, `install_args`, `install_targets`, `install_prefix_var` |
| `make_package` | Same as `autotools_package`, but `skip_configure = True` by default |
| `binary_package` | `install_script` (also auto-selected if you pass `src_compile` / `src_install` to any wrapper) |
| `cmake_package` | `source_subdir`, `cmake_args`, `cmake_defines`, `cmake_dep_defines`, `make_args` |
| `meson_package` | `meson_args`, `meson_defines`, `source_subdir`, `make_args` |
| `cargo_package` | `features`, `cargo_args`, `bins`, `vendor_deps` |
| `go_package` | `go_args`, `ldflags`, `bins`, `packages`, `vendor_deps`, `lib_only` |
| `python_package` | `use_setup_py`, `pip_args` |
| `perl_package` | `pre_build_cmds` |
| `mozbuild_package` | `mozconfig_options` |

### Auto-generated intermediate targets

Each call to a wrapper materializes a small chain of targets. All of them are
independently buildable:

- `:name-archive` — `http_file` (or `export_file` in vendor mode) for the
  downloaded source archive.
- `:name-src` — extracted source tree (`extract_source` rule).
- `:name-build` — output of the build rule (`<lang>_build`).
- `:name-stripped`, `:name-stamped`, `:name-signed` — created when the
  corresponding transform is applied.
- `:name` — `alias` pointing at the last target in the chain
  (`name-signed` → `name-stamped` → `name-stripped` → `name-build`).

Cargo and Go packages may additionally produce `:name-vendor-archive` /
`:name-vendor-src` when the vendor archive is mirror-hosted (see
[Mirror & seed configuration](#mirror--seed-configuration)).

### Build phases

`package()` dispatches into per-build-system rules that run these phases
in a single cacheable action (see `defs/rules/_common.bzl`):

1. **extract** — unpack `:name-archive` into `:name-src`.
2. **src_prepare** — apply `patches` (then `pre_configure_cmds`, if any).
3. **src_configure** — run `./configure` / `cmake` / `meson setup`.
4. **src_compile** — run `make` / `ninja` / `cargo build` / `go build`.
5. **src_install** — `make install DESTDIR=...` (or rule-specific
   equivalent); finally runs `post_install_cmds` against the prefix.

There is no separate `pre_configure` phase or `src_test` phase; per-package
shell hooks live in `pre_configure_cmds`, `pre_build_cmds`, and
`post_install_cmds`.

### Security features

- **Network isolation**: build phases run without network access using
  `unshare --net` (downloads run in their own `http_file` action).
- **Checksum verification**: SHA256 is required for every download.
- **GPG signature verification**: optional, configured per package.

## USE flags

USE flags drive conditional dependencies, configure arguments, cargo
features, and transforms. The flag → `select()` plumbing lives in
`defs/use_helpers.bzl`, but most BUCK files don't need to call those helpers
directly — `package()` (and therefore every wrapper) exposes the
`use_deps`, `use_configure`, `use_features`, and `use_transforms` kwargs and
expands them internally.

```python
load("//defs/packages:autotools.bzl", "autotools_package")

autotools_package(
    name = "curl",
    version = "8.8.0",
    url = "...",
    sha256 = "...",
    use_deps = {
        "ssl":  ["//packages/linux/system/libs/crypto/openssl:openssl"],
        "http2": ["//packages/linux/system/libs/net/nghttp2:nghttp2"],
    },
    use_configure = {
        "ssl":   ("--with-ssl",   "--without-ssl"),
        "http2": ("--with-nghttp2", "--without-nghttp2"),
    },
)
```

`use_deps` values may be a single target string, a list, or a 2-tuple
`(on_dep, off_dep)`. `use_configure` values may be a string (on-only), a
list, or a 2-tuple `(on_arg, off_arg)`. The flag also becomes a label
(`buckos:iuse:<flag>`) on the build target and an env var
(`USE_<FLAG>=1/0`) inside install scripts.

When you really need to compose `select()`s by hand, the helpers in
`defs/use_helpers.bzl` are: `use_bool`, `use_dep`, `use_configure_arg`,
`use_feature`, `use_expand_select`, `use_expand_dep`,
`use_expand_multi_deps`, `use_versioned_dep`.

## Transform chain

Pass `transforms` (a list) for transforms that run unconditionally, and
`use_transforms` (a dict) for transforms gated on USE flags. Transforms run
in order; the alias at `:name` points at the last one.

```python
autotools_package(
    name = "openssl-3.6",
    ...
    transforms = ["strip", "stamp"],   # always run
    use_transforms = {"ima": "ima"},   # run iff USE=ima
)
```

Supported transform names: `"strip"`, `"stamp"`, `"ima"` (see
`defs/rules/transforms.bzl`). USE-gated transforms always exist in the
graph but become zero-cost passthroughs when the flag is off.

## Multi-version packages

Multi-version packages are written as plain side-by-side `package()` calls
with an `alias` selecting the default. Example —
`packages/linux/system/libs/crypto/openssl/BUCK`:

```python
load("//defs/packages:autotools.bzl", "autotools_package")

autotools_package(name = "openssl-3.6", version = "3.6.1", ...)
autotools_package(name = "openssl-3.3", version = "3.3.2", ...)

alias(name = "openssl", actual = ":openssl-3.6")
```

Consumers reference the specific version (`:openssl-3.3`) when they need
it, or the alias (`:openssl`) for the default. For USE-flag-driven version
selection, use `use_versioned_dep()` from `defs/use_helpers.bzl`.

## Private patch registry

A private registry lets you apply downstream patches and per-package
configure/cflag overrides without touching the upstream BUCK files.

1. Create `patches/registry.bzl` (gitignored) with:

   ```python
   PATCH_REGISTRY = {
       "musl": {
           "patches":              ["//patches:musl-fix.patch"],
           "extra_configure_args": ["--enable-debug"],
           "extra_cflags":         ["-DCUSTOM"],
       },
   }
   ```

2. Export the patch files in `patches/BUCK` via `export_file()`.
3. Change the load line at the top of `defs/package.bzl` from
   `load("//defs:empty_registry.bzl", "PATCH_REGISTRY")` to
   `load("//patches:registry.bzl", "PATCH_REGISTRY")`.

`package()` merges public `patches` / `configure_args` / `extra_cflags`
first, then the private registry entries. Only `patches`,
`extra_configure_args`, and `extra_cflags` are honored — there is no
per-package `env` override field.

## Auto-injected host tools

Every call to a configurable build rule (`autotools`, `make`, `meson`,
`cmake`, `mozbuild`) automatically gets a baseline of host tools injected
into its dep set so configure scripts and Makefiles never reach into the
host `/usr/bin`. The injected set (see `defs/package.bzl` ~ lines 399-460):

- POSIX: `bash`, `coreutils`, `findutils`, `sed`, `gawk`, `grep`,
  `diffutils`, `patch`, `tar`, `gzip`, `xz`, `bzip2`.
- Interpreters: `python-host`, `perl`.
- Build tools: `m4`, `make`, `pkg-config`.
- `meson`/`mozbuild` also get `meson` and `ninja`.
- `cmake` also gets `cmake` and `ninja`.
- `ccache` is added when `[buckos.cache] mode = enabled`.

To break cycles (each of these tools is itself a package that uses
`package()`), `_TOOL_BLOCKLIST` in `defs/package.bzl` lists package names
that must not get the injection: the tools themselves and their transitive
build-time deps (`zlib`, `expat`, `libffi`, `ncurses`, `readline`, etc.).
If you see a "cyclic dependency" failure on a build tool, that name
probably needs to be added to `_TOOL_BLOCKLIST`.

With a prebuilt seed (`[buckos] seed_path` or `seed_url` set), the seed's
hermetic PATH provides these tools and the explicit exec_deps are dropped.

## Mirror & seed configuration

All read at module load by `defs/package.bzl`. Set in `.buckconfig`:

```ini
[mirror]
mode      = upstream        ; or "vendor"
base_url  =                 ; optional fast mirror prepended to upstream URL
prefix    =                 ; full URL prefix used for content-addressed downloads
params    =                 ; trailing query string appended to mirror URLs
vendor_dir =                ; directory of pre-downloaded archives (vendor mode)

[buckos]
seed_path = /path/to/seed.tar.zst   ; OR
seed_url  = https://example/seed.tar.zst
default_toolchain = buckos//tc/seed:seed-toolchain

[buckos.cache]
mode         = enabled       ; "enabled" or "disabled"
location     = homedir       ; "homedir" → ~/.buckos/caches/ccache
                             ; "projectdir" → .buckos/cache/ccache
ccache_size  = 100G
sccache_size = 100G
```

Mode summary:

- **`mirror.mode = upstream`** (default): downloads come from the upstream
  URL, optionally with `base_url` prepended as a faster mirror.
- **`mirror.mode = vendor`**: `:name-archive` is an `export_file` from
  `vendor_dir/<filename>`; `cargo_package` / `go_package` calls also auto-wire
  their `vendor_deps` from the same directory.
- **`prefix` set**: downloads use a content-addressed URL
  `<prefix>/<n>/<name>-<version>-<sha12>.<ext><params>`.
- **Seed mode** (`seed_path` or `seed_url` set): host tools are provided
  by the seed's hermetic PATH; in this mode auto-injected exec_deps are
  dropped, and explicit `host_deps` that the seed already ships are pruned.
- **Source mode** (no seed configured): host tools are built from source.

## Kernel and rootfs rules

Kernel and rootfs rules are imported directly from `defs/rules/`, not
through `package()`:

```python
load(
    "//defs/rules:kernel.bzl",
    "kernel_build",
    "kernel_config",
    "kernel_headers",
    "kernel_btf_headers",
    "kernel_modules_install",
)
load("//defs/rules:rootfs.bzl", "rootfs")
```

`kernel_build` supports `patches` and an external `modules` list for
out-of-tree module sources. See `packages/linux/kernel/buckos-kernel/BUCK`
and `packages/linux/system/BUCK` for representative usage.

## Specifications

Formal specifications live in `specs/` with YAML frontmatter. Only specs
with `status = approved` should be used as implementation references. The
human entry point is `specs/INDEX.md`; the machine-readable index is
`specs/REGISTRY.json`.
