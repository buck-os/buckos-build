# Fast iteration playbook

Feedback loops for reproducing known failure modes without waiting on
the full test suite.  Tiered by cost ‚Äî run the cheapest tier that
reproduces the failure class you're investigating.

All buck2 commands below are cell-relative (`//...`) and must be run
from the buckos project root ‚Äî the directory containing `.buckconfig`
with the cell aliases for `//` to resolve.  Running from a sibling
project or an integrated-tree root gives cell-resolution errors.

> **Status note.** Some of the failure-class names below (kernel exit
> -11, `-latomic`, iconv/msgfmt) describe historical regressions tied
> to earlier iterations of the dep-RPATH / sysroot handling in
> `tools/_env.py`.  The targets remain valid as fast smoke tests ‚Äî
> they exercise the most fragile cross-package interactions ‚Äî but the
> exact stack traces and helper-function names from those original
> incidents have moved as the helper code has evolved.  Use the tiers
> as a workflow guide; cross-reference the current code in
> `tools/_env.py`, `tools/kernel_build.py`, and `tools/rewrite_interps.py`
> when interpreting any specific failure.

## Tier 1 ‚Äî kernel smoke test (~2 min incremental)

The `test-kernel` fixture is the smallest cross-package action sequence
that exercises kernel build + the toolchain's sysroot-glibc handling.
It's the canonical first-thing-to-run when a build helper change might
have broken dep-RPATH, sysroot lib resolution, or shebang rewriting.

```sh
buck2 build '//tests/fixtures:test-kernel' --no-remote-cache
```

- **Expected runtime**: ~2 min incremental on warm cache, ~10 min cold.
- **What to look for**: any non-zero exit from the action ‚Äî in
  particular signal exits (negative return codes from `_run()` in
  `tools/kernel_build.py:503`) usually indicate ABI mismatch between
  the host loader and a sysroot-linked binary.
- **What this validates**: the most common cross-action failure mode ‚Äî
  one action writing into a shared dep prefix and a later action
  picking up the result.

To force a clean exercise of first-time setup, `buck2 clean` first; to
exercise cache-reuse paths, leave the prior build outputs in place.

## Tier 2 ‚Äî per-failure single-package builds

Each target reproduces a distinct sysroot / dep-RPATH failure class
without running the full ISO assembly or VM boot.  Run with
`--no-remote-cache` to force a local exercise of the toolchain wrappers.

| Failure class | Target | Typical error pattern |
|--------|--------|---------------|
| kernel build | `//packages/linux/kernel/buckos-kernel:buckos-kernel-minimal` | signal/abort from `make` or in-tree host tools |
| `-latomic` link | `//packages/linux/system/libs/numactl:numactl-build` | `/usr/bin/ld: cannot find -latomic` |
| iconv/msgfmt | `//packages/linux/lang/binutils:binutils-build` | `msgfmt: ... iconv` error |
| iconv (alt) | `//packages/linux/core/glib:glib-build` | `msgfmt: ... iconv` error |

Expected per-target runtime ranges ~1‚Äì5 min incremental.

The `-build` suffix is auto-generated by `defs/package.bzl` for every
package declared via `package()` ‚Äî see the `build_target = name + "-build"`
line at `defs/package.bzl:641` (auto-created sibling targets are listed
near the top of that file: `-archive`, `-src`, `-build`, optionally
`-stripped` / `-stamped` / `-signed`, and a `:name` alias).

Tier 2 only reproduces the failures that surface inside a single
package action.  Cross-action interactions (where action A's output
poisons action B's view of a shared dep) require Tier 1's
multi-package sequence.

## Tier 3 ‚Äî single-action replay (<30 sec)

When you have a specific failing action and want to iterate on the
helper code path without re-running the full build, extract the
action from the CAS and run its `./exe` locally.

```sh
# 1. Identify the failing action from the most recent build.
buck2 log what-failed

# 2. Download the action's inputs + command wrapper from CAS.
frecli cas download-action <digest>:<size>

# 3. Run it locally and iterate.
cd <downloaded_dir>
./exe
```

- **Expected runtime**: <30 sec per replay, because the action runs
  only its own subprocess ‚Äî not any upstream deps.
- **Workflow**: edit the helper under `tools/` that the action calls
  (e.g. `tools/_env.py`, `tools/kernel_build.py`,
  `tools/rewrite_interps.py`), re-run `./exe`, check output.  No buck2
  round-trip.
- **Validates**: targeted helper changes without waiting on buck2's
  action graph.

The CAS download preserves the exact input state the action saw, so a
local re-run reproduces the failure faithfully ‚Äî modulo glibc /
loader differences between the replay host and the original worker
(see Gotchas).

## Recommended diagnostic workflow

1. Run the Tier 1 or Tier 2 command that matches the failure class.
2. On failure: `buck2 log what-failed` to get the action digest.
3. `frecli cas download-action <digest>:<size>` to extract it.
4. `cd <downloaded_dir> && ./exe` to reproduce locally in <30 sec.
5. Edit the relevant helper under `tools/`, re-run `./exe`, verify.
6. Re-run the original Tier 1 / Tier 2 build command to confirm the
   fix.

This keeps the edit-verify cycle under 30 sec once a failing action
is extracted, versus a full suite run.

## What these tiers do NOT cover

Tier pass is not a ship signal ‚Äî each tier has specific blind spots.

- **Tier 1/2 local builds vs RE workers**: local builds don't
  exercise RE-only syscall restrictions, RE network topology, or
  RE-worker scheduling.  A fix that passes locally can still fail
  on RE.
- **Tier 3 replay host glibc**: `./exe` runs against the REPLAY
  host's `/bin/sh` and its glibc, which may differ from the RE
  worker's glibc.  If the failure is sensitive to glibc version,
  Tier 3 pass does not imply RE-safe.
- **Tier 2 cross-package interactions**: per-package builds don't
  exercise compile-time interactions between siblings.  Tier 1
  `test-kernel` covers the multi-action sequence.
- **Tier 1 first-run vs cache-reuse**: a clean build only exercises
  first-time setup.  Cache-invalidation bugs need stale state on
  disk to trigger.

## Map of failure modes to likely fix sites

| Failure class | First place to look |
|---------------|----------------------|
| kernel signal exit | `tools/_env.py` `sysroot_lib_paths` / `derive_lib_paths` (line 299, 323) ‚Äî sysroot vs host loader mismatch usually shows here |
| `-latomic` link failure | `tools/_env.py` `filter_path_flags` (line 373) and the seed-toolchain copy path in `tools/rewrite_interps.py` |
| iconv/msgfmt `GCONV_PATH` | `tools/_env.py` `derive_lib_paths` ‚Äî add a sysroot gconv-modules probe fallback so consumers without a local gconv inherit one from the sysroot |

These mappings are starting points, not guarantees.  When in doubt,
attach a debugger to the failing `./exe` from Tier 3 and trace which
helper produced the bad env / argv.

## Gotchas

- `buck2 build --no-remote-cache` bypasses RE cache but still uses
  the local artifact cache.  For full isolation use `buck2 clean`
  first; for action-level isolation Tier 3 is more targeted.
- `buck2 log what-failed` only reports the most recent failure ‚Äî if
  your build has several, use `buck2 log show` to page through.
- `frecli cas download-action` needs a `<digest>:<size>` pair; copy
  both fields from `buck2 log what-failed` output.  These hashes are
  input-content-hashed ‚Äî editing a helper changes the hash, so the
  digest from `what-failed` is valid only for that specific build
  run.  After any edit, re-run Tier 1 / Tier 2 to regenerate a fresh
  digest.
- The Buck2 daemon can take 90s+ to become ready after `.bzl` edits.
  Set `BUCKD_STARTUP_TIMEOUT=600` (see MEMORY.md) instead of killing
  the daemon ‚Äî restarts are expensive.
- All buck2 commands assume cwd is the buckos project root (where
  the cell aliases for `//` resolve).  Running from a sibling project
  or an integrated-tree root gives cell-resolution errors.

## Adding a new entry

When a new failure mode surfaces, add a Tier 2 row with:
- a target that reproduces it in <5 min,
- the exact grep-able error string,
- a pointer to the helper function in `tools/_env.py` or a sibling
  helper where the fix is likely to land.
